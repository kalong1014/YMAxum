## 一、文件及目录命名规则

### 1. 通用命名规范
- **目录命名**：采用小写字母，单词间用下划线分隔（snake_case）
- **文件命名**：
  - 源代码文件：小写字母，单词间用下划线分隔（如app_state.rs）
  - 核心模块文件：按功能命名（如state.rs）
  - 插件文件：包含插件名称（如customer_service.rs）
  - 测试文件：以_test.rs结尾（如state_test.rs）
  - 脚本文件：Linux(.sh)、Windows(.bat)结尾
- **配置文件**：采用TOML格式，以.toml结尾
- **文档文件**：采用Markdown格式，以.md结尾

### 2. 核心代码目录命名规范
- **src/core/**：核心功能模块
- **src/command/**：命令解析引擎
- **src/security/**：安全防护模块
- **src/ops/**：运维管理模块
- **src/scene/**：场景适配代码
- **src/guf/**：GUF集成模块

### 3. 插件目录命名规范
- **plugins/插件名称/**：每个插件独立子目录
- **插件内部结构**：
  - src/：插件源代码
  - Cargo.toml：插件依赖配置
  - manifest.json：插件元数据
  - config.toml：插件配置

### 4. 配置文件命名规范
- **config/模块名.toml**：模块配置文件
- **环境配置**：environment.toml
- **数据库配置**：database.toml
- **安全配置**：security.toml

## 三、各目录核心用途

### 1. 根目录
- **Cargo.toml**：项目依赖配置，定义了项目的依赖包和版本
- **README.md**：项目基本信息，包含项目介绍、使用方法等
- **Dockerfile**：Docker构建配置，用于容器化部署
- **Makefile**：构建脚本，定义了常用的构建和测试命令
- **构建脚本**：各平台的构建脚本，用于生成可执行文件

### 2. src/ 目录
- **src/core/**：核心功能模块，包含状态管理、API迭代、中间件等核心组件
- **src/command/**：命令解析引擎，处理命令行命令的解析和执行
- **src/security/**：安全防护模块，包含加密、认证、授权等安全功能
- **src/ops/**：运维管理模块，包含配置热更新、健康检查等运维功能
- **src/scene/**：场景适配代码，包含各行业场景的适配器
- **src/guf/**：GUF集成模块，处理与Godot UI Framework的集成
- **src/plugin/**：插件系统，管理插件的加载、生命周期等
- **src/ai/**：AI相关功能，包含智能路由、模型管理等
- **src/bigdata/**：大数据处理，包含数据采集、分析、存储等
- **src/blockchain/**：区块链集成，包含智能合约、身份管理等
- **src/cloud/**：云服务集成，包含Kubernetes、服务网格等

### 3. plugins/ 目录
- **plugins/analytics/**：分析插件，提供数据统计和分析功能
- **plugins/auth/**：认证插件，提供用户认证和授权功能
- **plugins/cache/**：缓存插件，提供缓存管理功能
- **plugins/monitoring/**：监控插件，提供系统监控功能
- **plugins/payment/**：支付插件，提供支付集成功能
- **plugins/示例目录**：各语言插件示例，用于开发者参考

### 4. config/ 目录
- **config/database.toml**：数据库配置，定义数据库连接信息
- **config/environment.toml**：环境配置，定义环境变量和全局设置
- **config/security.toml**：安全配置，定义安全相关设置
- **config/plugin.toml**：插件配置，定义插件加载和管理设置
- **config/server.toml**：服务器配置，定义HTTP服务器设置

### 5. templates/ 目录
- 存放HTML模板文件，用于生成Web界面
- 包含聊天、防欺诈、权限管理等各种功能的模板

### 6. tests/ 目录
- **tests/api_tests.rs**：API测试，测试API接口的功能
- **tests/integration_tests.rs**：集成测试，测试系统各组件的集成
- **tests/e2e_tests.rs**：端到端测试，测试整个系统的功能
- **tests/guf_integration_tests.rs**：GUF集成测试，测试与GUF的集成

### 7. scripts/ 目录
- 存放自动化脚本，用于代码质量检查、依赖管理、部署等
- 包含各种自动化工具，提高开发效率

### 8. code_quality_results/ 目录
- 存放代码质量分析结果，包含代码复杂度、重复代码等分析

### 9. performance_results/ 目录
- 存放性能测试结果，包含基准测试报告和结果数据

### 10. deployments/ 目录
- 存放部署文件，包含各环境的可执行文件和配置

### 11. third_party/ 目录
- **用途**：存放第三方开源组件原码，用于软著申请时明确区分自研代码和开源代码
- **结构**：
  - third_party/axum/：axum 原码（MIT 协议）
  - third_party/godot_guf/：Godot GUF 原码（对应开源协议）
  - third_party/other/：其他开源组件
- **软著申请用途**：在软著申请时，third_party 目录下的代码不纳入自研代码统计，仅作为开源合规性证明

## 四、专属目录规范

### 1. models/ 目录
- **用途**：存放数据模型定义
- **结构**：
  - models/模型名.rs：模型定义
  - 包含结构体定义、序列化/反序列化等
- **命名规范**：采用PascalCase命名法

### 2. data/ 目录
- **用途**：存放数据文件
- **结构**：
  - data/原始数据/：原始数据文件
  - data/处理后数据/：处理后的数据文件
- **格式规范**：
  - 文本数据：UTF-8编码
  - 结构化数据：JSON、TOML格式
  - 二进制数据：适当的二进制格式

### 3. train/ 目录
- **用途**：存放模型训练相关文件
- **结构**：
  - train/训练脚本/：训练模型的脚本
  - train/模型文件/：训练后的模型文件
- **格式规范**：
  - 脚本文件：.rs、.py等
  - 模型文件：根据模型类型选择适当格式

### 4. infer/ 目录
- **用途**：存放推理相关文件
- **结构**：
  - infer/推理脚本/：模型推理的脚本
  - infer/配置文件/：推理配置文件
- **格式规范**：
  - 脚本文件：.rs、.py等
  - 配置文件：TOML格式

### 3. 变更管理流程
1. **变更申请**：提交变更申请，说明变更原因和影响
2. **变更评估**：评估变更的可行性和影响范围
3. **变更实施**：按照评估结果实施变更
4. **变更验证**：验证变更是否达到预期效果
5. **变更记录**：记录变更信息，更新目录变更记录

## 七、目录规范执行要求

1. **严格遵守命名规范**：所有目录和文件必须按照规定的命名规范命名
2. **保持目录结构清晰**：定期清理无用目录和文件，保持目录结构清晰
3. **目录用途明确**：每个目录的用途必须明确，避免功能混淆
4. **变更记录完整**：所有目录变更必须记录在变更记录中
5. **定期审查**：定期审查目录结构，确保符合规范要求

通过严格执行目录规范，可以提高代码的可维护性和可读性，便于团队协作和项目管理。
