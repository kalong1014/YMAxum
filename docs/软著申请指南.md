# 软著申请指南

## 一、软著分层布局方案

### 1. 方案概述
基于 2026 年软著审核规则，YMAxum 项目采用分层布局策略，将代码分为核心层、开源适配层、业务扩展层和工具链层，每一层都可以单独申请软著，既满足自研占比要求，又能最大化获取政策补贴。

### 2. 分层结构

| 分层        | 代码范围                          | 自研占比 | 软著申请要点                                                                 | 政策补贴方向                                                                 |
|-------------|-----------------------------------|----------|------------------------------------------------------------------------------|------------------------------------------------------------------------------|
| 核心层      | `src/core/` 全目录                | 100%     | 突出自研运行时、跨端桥接算法、核心 API 的原创性                              | 国家级/省级"核心技术攻关补贴""高新技术企业核心知识产权"                       |
| 开源适配层  | `src/guf/`、`src/adapter/`        | ≥85%     | 明确自研适配逻辑，提交开源组件使用声明                                      | 市级"开源生态适配补贴""中小企业技术改造补贴"                                |
| 业务扩展层  | `src/plugin/`、`src/scene/`        | 100%     | 突出认证/日志/监控的落地场景，提交应用案例                                  | 区级"产业化应用补贴""科技型中小企业研发费用补贴"                             |
| 工具链层    | `src/tools/`、`scripts/`           | 100%     | 突出开发和运维工具的创新性，提交工具使用效果证明                            | 区级"科技创新工具补贴""数字化转型工具补贴"                                   |

### 3. 目录结构调整
- **新增 `third_party/` 目录**：存放开源组件原码，用于明确区分自研代码和开源代码
  - `third_party/axum/`：axum 原码（MIT 协议）
  - `third_party/godot_guf/`：Godot GUF 原码（对应开源协议）
  - `third_party/other/`：其他开源组件

### 4. 版权声明规范
所有自研代码文件头部必须添加以下版权声明：
```rust
// Copyright © [企业全称信息以便后期替换] 2020-2026 YMAxum Framework. All Rights Reserved.
// 自研原创代码，未经授权禁止复制/修改
```

## 二、软著申请材料准备

### 1. 核心层申请材料
- **代码清单**：`src/core/` 目录下的所有自研代码文件
- **技术文档**：包含核心层的架构设计、创新点说明、功能模块描述
- **创新点说明**：重点说明自研运行时解决跨端性能瓶颈、自研中间件的细粒度权限控制等差异化创新
- **架构设计图**：核心层的系统架构图、模块关系图

### 2. 开源适配层申请材料
- **代码清单**：`src/guf/`、`src/adapter/` 目录下的自研适配代码
- **技术文档**：包含适配层的架构设计、适配逻辑说明
- **开源组件使用声明**：记录使用的开源组件版本、协议类型、用途
- **开源协议文件**：axum（MIT）、Godot GUF 等开源组件的协议文件

### 3. 业务扩展层申请材料
- **代码清单**：`src/plugin/`、`src/scene/` 目录下的自研代码
- **技术文档**：包含业务扩展层的功能模块描述、应用场景说明
- **应用案例**：配套合作企业的项目验收报告、试点场景运行数据
- **客户合作证明**：与客户的合作协议、项目实施证明

### 4. 工具链层申请材料
- **代码清单**：`src/tools/`、`scripts/` 目录下的自研代码
- **技术文档**：包含工具链的功能描述、使用方法说明
- **工具使用效果证明**：工具使用的性能提升数据、开发效率提升证明
- **用户反馈**：内部或外部用户的使用反馈

## 三、开源合规管理

### 1. 开源组件清单
| 组件名称 | 版本 | 协议类型 | 用途 | 来源 |
|---------|------|----------|------|------|
| axum | 0.7+ | MIT | Web 框架 | https://crates.io/crates/axum |
| Godot GUF | 4.x | MIT | UI 框架 | https://godotengine.org/ |
| Tokio | 1.x | MIT | 异步运行时 | https://crates.io/crates/tokio |
| Serde | 1.x | MIT | 序列化/反序列化 | https://crates.io/crates/serde |

### 2. 开源合规要求
- **协议备案**：将开源组件的协议文件提交软著登记机构和企业内部备案
- **使用审批**：留存开源组件使用的内部审批记录
- **风险排查**：每季度委托第三方审计开源组件使用合规性
- **修改记录**：如修改开源组件，需单独分支管理，标注修改内容并遵循开源协议

## 四、软著申请流程

### 1. 前置准备
- **代码审计**：委托第三方机构出具"自研占比+开源合规"审计报告
- **材料准备**：按照各分层要求准备申请材料
- **版本快照**：对申请的代码版本进行 Git 标签标记，留存快照

### 2. 申报顺序
1. **核心层**：100% 自研，最易通过，作为其他层的基础
2. **开源适配层**：基于核心层，明确开源边界
3. **业务扩展层**：基于核心层和适配层，突出落地场景
4. **工具链层**：基于整体框架，突出工具创新

### 3. 提交申请
- **在线提交**：通过中国版权保护中心官网提交软著申请
- **材料邮寄**：将纸质材料邮寄至中国版权保护中心
- **缴费**：按照规定缴纳软著登记费用

### 4. 审批流程
- **受理**：版权中心受理申请，发放受理通知书
- **审查**：版权中心审查申请材料，可能要求补充材料
- **发证**：审查通过后，发放软著登记证书

## 五、基于框架开发的项目适配方案

### 1. 适配步骤
1. **目录结构调整**：在项目根目录创建 `third_party/` 目录，存放使用的开源组件
2. **版权声明添加**：为所有自研代码文件添加版权声明
3. **分层申请**：根据项目特点，选择合适的分层进行软著申请
4. **材料准备**：按照本指南的要求准备申请材料

### 2. 适配示例
- **企业管理系统**：可申请核心层（基础框架）+ 业务扩展层（企业管理功能）
- **电商平台**：可申请核心层（基础框架）+ 业务扩展层（电商功能）+ 工具链层（数据分析工具）
- **物联网系统**：可申请核心层（基础框架）+ 开源适配层（物联网协议适配）+ 业务扩展层（物联网管理功能）

### 3. 注意事项
- **代码边界**：明确区分自研代码和开源代码，避免开源代码混入自研目录
- **版权声明**：确保所有自研代码文件都添加了版权声明
- **开源合规**：使用的开源组件必须符合其开源协议要求
- **版本管理**：对申请的代码版本进行标记，便于后续维护和更新

## 六、政策补贴申请指南

### 1. 补贴类型
- **国家级**：核心技术攻关补贴、高新技术企业认定、国家重点研发计划
- **省级**：科技创新专项资金、软件产业发展专项资金、中小企业技术创新基金
- **市级**：软件企业认定、数字化转型补贴、科技创新券
- **区级**：科技型中小企业研发费用补贴、产业化应用补贴、创业扶持资金

### 2. 补贴申请要点
- **核心层**：申报高等级补贴，突出核心技术创新
- **适配层**：申报市级补贴，强调开源生态适配
- **扩展层**：申报区级补贴，突出产业化应用
- **工具链层**：申报区级补贴，强调工具创新

### 3. 材料准备
- **软著证书**：已获得的软著登记证书
- **技术查新报告**：第三方机构出具的技术查新报告
- **项目可行性报告**：详细说明项目的技术可行性、市场前景
- **财务报表**：企业近三年的财务报表
- **其他证明**：专利证书、获奖证明、客户合作证明等

## 七、软著维护与更新

### 1. 维护要求
- **年度更新**：每年更新软著，如核心层迭代、适配层新增开源组件适配
- **版本管理**：对每次更新的代码版本进行 Git 标签标记
- **文档更新**：及时更新技术文档，反映代码的变化

### 2. 更新流程
- **代码审计**：对更新的代码进行自研占比和开源合规审计
- **材料准备**：准备更新后的申请材料
- **提交申请**：向版权中心提交软著更新申请
- **证书更新**：获取更新后的软著登记证书

## 八、常见问题解答

### 1. 软著申请需要多长时间？
- 一般情况下，软著申请需要 30-60 个工作日。
- 加急申请可以缩短至 1-15 个工作日，需缴纳加急费用。

### 2. 软著申请的费用是多少？
- 普通申请：300 元/件
- 加急申请：根据加急程度，费用在 600-2500 元/件之间

### 3. 软著申请需要提供源代码吗？
- 需要提供源代码的前 30 页和后 30 页，每页不少于 50 行。
- 代码量较少的项目需要提供全部源代码。

### 4. 开源组件如何处理？
- 开源组件应单独存放在 `third_party/` 目录
- 申请软著时，开源组件不纳入自研代码统计
- 需要提供开源组件的使用声明和协议文件

### 5. 软著证书有什么用？
- 作为软件著作权的法律证明
- 用于申请高新技术企业认定
- 用于申请各种政策补贴
- 用于软件产品登记
- 用于维权和诉讼
